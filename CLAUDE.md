## 0. 핵심 운영 원칙

Claude Code는 **빠르게 작업하되, 위험한 수정은 반드시 멈추고 확인**한다.

### 기본 원칙
- 요청받은 부분만 수정한다.
- 임의 리팩토링, 임의 디자인 변경, 임의 문구 변경 금지.
- 불명확하면 추측해서 수정하지 말고 먼저 질문한다.
- `server.js`, Supabase/DB, 로그인, 저장, 삭제, 관리자 기능, 배포 관련 작업은 신중하게 처리한다.
- 큰 수정이 아니면 과도한 절차로 시간을 끌지 않는다.

## 1. 작업 시작 전 규칙

모든 작업을 시작할 때 매번 긴 보고서를 작성하지 않는다.  
작업 성격에 따라 **기본 작업**과 **확인 필요 작업**으로 나눈다.

### 1-1. 기본 작업 — 바로 진행 가능

아래 작업은 사용자 확인 없이 바로 진행해도 된다.

- 오타 수정
- 단순 문구 수정
- 작은 CSS/UI 조정
- 한 파일 안에서 끝나는 간단한 버그 수정
- 주석 정리
- console 로그 제거
- 명확하게 요청받은 작은 수정

단, 수정 후에는 간단히 보고한다.

```text
수정 파일:
변경 내용:
확인한 내용:
```

### 1-2. 확인 필요 작업 — 수정 전 반드시 멈춤

아래 작업은 파일을 수정하기 전에 먼저 사용자에게 확인받는다.

- 여러 파일을 동시에 수정하는 작업
- `server.js` 수정
- Supabase / DB 구조 변경
- 데이터 저장, 삭제, 수정 로직 변경
- 로그인 / 회원 / 권한 관련 수정
- 관리자 페이지 기능 수정
- 결제, 견적, 주문, 상담 저장 등 핵심 기능 수정
- 기존 디자인을 크게 바꾸는 작업
- 기능 구조를 바꾸는 리팩토링
- 배포 명령 실행
- 데이터 삭제 또는 초기화
- 사용자가 “먼저 확인해”, “계획부터 말해”, “어디 고칠지 먼저 알려줘”라고 말한 작업

이 경우 아래 형식으로 짧게 보고한다.

```text
요청 요약:
수정 예상 파일:
수정 범위:
건드리지 않을 부분:
백업 필요 여부:
진행해도 될까요?
```

사용자가 “진행”이라고 답하기 전까지 파일 수정 금지.

## 3. team-orchestrator 실행 기준

HTML/CSS/JS를 수정했다고 해서 매번 실행하지 않는다.  
아래 경우에만 `@team-orchestrator`를 실행한다.

### 실행해야 하는 경우
- 새 기능 추가
- 여러 파일 수정
- 배포 전 최종 점검
- 로그인 / DB / 저장 / 삭제 / 관리자 기능 수정
- 화면 흐름이나 상태 관리가 바뀌는 작업
- 사용자가 명시적으로 검토를 요청한 경우

### 실행하지 않아도 되는 경우
- 오타 수정
- 단순 문구 수정
- 작은 CSS 조정
- 한 파일 안에서 끝나는 간단한 버그 수정
- 주석 수정
- console 로그 제거

실행해야 하는 경우에는 수정 직후 아래처럼 진행한다.

```text
@team-orchestrator 방금 수정한 내용 검토해줘.
```

## 4. 작업 완료 후 보고

작업 완료 후에는 길게 설명하지 말고 아래 형식으로 보고한다.

```text
완료했습니다.

요청 내용:
- 

원인 / 이유:
- 

해결 방법:
- 

결과: 성공 / 실패

수정 파일:
- 

추가 확인 필요:
- 없음 / 있음
```

실패한 경우에는 결과를 "실패"로 표시하고 이유를 간단히 적는다.

배포가 필요한 경우에는 반드시 별도로 말한다.

```text
배포가 필요합니다.
```

배포가 필요 없으면:

```text
배포 없이 로컬 수정만 완료했습니다.
```

## 6. 에이전트 팀 구성

이 프로젝트는 5개의 전문 에이전트로 구성된 팀을 사용합니다.

| 에이전트 | 역할 |
|---------|------|
| `team-orchestrator` | 헤드 에이전트 - 팀 전체 조율 |
| `html-css-js-reviewer` | HTML/CSS/JS 코드 품질 검토 |
| `web-security-auditor` | 보안 취약점 검사 |
| `code-bug-fixer` | 버그 탐지 및 수정 |
| `frontend-test-validator` | 테스트 검증 |

---

## 7. 에이전트 사용 기준

### 팀 전체 검토가 필요한 경우 → `@team-orchestrator`
- 새 기능 완성 후
- 여러 파일을 함께 수정한 후
- 배포 전 최종 검토
- 코드 전반적인 품질 점검이 필요할 때

### 개별 에이전트 사용
- 코드 스타일만 확인 → `@html-css-js-reviewer`
- 보안만 빠르게 확인 → `@web-security-auditor`
- 버그만 확인 → `@code-bug-fixer`
- 테스트만 확인 → `@frontend-test-validator`

| `04_할인규칙.md` | 신규아파트 10% 할인 조건 |
| `05_견적표원칙.md` | 견적서 출력 원칙 |
| `06_상담흐름.md` | 상담 진행 순서 |
